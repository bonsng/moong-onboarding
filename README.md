# 뭉(Moong) 온보딩 랜딩 페이지

반려견 가계부 서비스 뭉(Moong)의 랜딩 페이지입니다.
이 프로젝트는 **바이브 코딩(Vibe Coding)**으로 제작되었으며, 아래에 그 과정을 기록합니다.

> **배포 주소**: [landing.moongmoong.site](https://landing.moongmoong.site)

## 기술 스택

- Next.js 16 / React 19
- Three.js + React Three Fiber (3D 시바견 모델)
- Framer Motion (스크롤 애니메이션)
- Tailwind CSS v4
- Lucide React (아이콘)

---

## 바이브 코딩 과정

### 1단계: 기획 문서 → 프롬프트 설계

팀에서 작성한 온보딩 기획 문서를 기반으로, 원하는 결과물의 방향을 프롬프트에 담았습니다.

아래는 프롬프트에 첨부한 기획 문서(`landing.md`)의 내용입니다:

이 기획 문서와 함께 프롬프트에 포함한 추가 요구사항:
- 각 섹션에 들어갈 앱 스크린샷 이미지 경로
- 원하는 분위기 (따뜻한 톤, 반려견 테마, 크림색 배경)
- 기술적 요구사항 (Next.js, Tailwind, Framer Motion 애니메이션, 3D 강아지 모델)
- 8개 섹션 구성: 랜딩 히어로 → 앱 소개 → 가계부 → AI 예측 → 진료비 → 저금통 → 월간 레포트 → CTA

**핵심은 "한 번에 전체 구조를 잡는 것"이었습니다.**
섹션 하나씩 만들어달라고 하지 않고, 기획 문서 전체를 맥락으로 제공하고 완성된 랜딩 페이지를 요청했습니다.

### 2단계: 초기 생성 결과

Claude Code가 **43개 파일, 약 9,500줄**의 코드를 한 번에 생성했습니다.

```
src/
├── app/                    # Next.js 앱 라우터
├── components/
│   ├── sections/           # 8개 섹션 컴포넌트
│   └── ui/                 # 공통 UI (3D 모델, 애니메이션, 네비게이션)
├── hooks/                  # 스크롤 추적 훅
└── lib/                    # 애니메이션 variants, 상수
```

초기 결과물을 확인한 뒤, 추가 프롬프트를 통해 다음 기능들을 덧붙였습니다:
- Three.js 를 통해 구현한 3D 강아지 모델 애니메이션 (달리기 → 앉기 → 헥헥거리기 애니메이션)
- GLB 모델을 불러오는 **ShibaModel** 컴포넌트
- 스크롤에 반응하는 **발자국 트레일** 사이드바
- 섹션별 진입 애니메이션 (fade, scale, stagger 등 10종)
- Intersection Observer 기반 **도트 네비게이션**

### 3단계: 검증과 수정

초기 결과물을 브라우저에서 확인하며 다음과 같은 문제를 발견하고 수정했습니다.

| 날짜 | 수정 내용 | 발견 방법 |
|------|-----------|-----------|
| 2/23 | 제목 줄넘김 정리, 폰트 변경, 배경 효과 추가 | 육안 검수 - 모바일/데스크톱 레이아웃 확인 |
| 2/23 | 스크린샷 이미지 교체 및 최적화 | 실제 앱 스크린샷으로 교체 필요 |
| 2/25 | 스크린샷 이미지 영역 확대 | 핵심 콘텐츠인 앱 화면이 너무 작게 보이는 문제 |
| 2/25 | 이모지 → lucide-react 아이콘 교체 | OS별 이모지 렌더링 불일치 문제 발견 |
| 2/25 | 불필요한 SVG 장식 요소 제거 | 시각적 노이즈 판단 |

**검증 방식**:
- 브라우저에서 직접 스크롤하며 애니메이션 동작 확인
- 모바일 뷰포트에서 레이아웃 깨짐 확인
- 크로스 브라우저(Chrome, Safari)에서 이모지/폰트 렌더링 차이 발견

## 회고

### 잘 된 점

- **실제로 팀원간의 합의를 통해 적은 기획 문서를 같이 넣은 것이 효과적이였다.**
  섹션 간의 톤, 흐름, 색감이 일관성 있게 나왔다. 내용 생성까지 맡기면 서비스의 의도와 달라졌을것이다.

- **초기 생성 품질이 예상보다 높았다.**
  3D 모델, 스크롤 애니메이션, 반응형 레이아웃까지 한 번에 동작하는 결과물이 나왔다.

- **수정 작업이 "처음부터 만들기"가 아니라 "다듬기"였다.**
  전체 6커밋 중 첫 커밋이 전체의 95% 이상이었고, 이후 커밋은 모두 미세 조정이었다.

- **계획 모드(Plan Mode)를 적극적으로 활용한 것이 도움이 됐다.**
  코드를 바로 생성하기 전에 계획 모드로 전체 구조와 섹션 구성을 먼저 설계했다. 덕분에 AI가 전체적인 방향을 잡은 상태에서 코드를 작성해, 섹션 간 일관성이 높고 되돌리는 작업이 적었다.

### 아쉬운 점

- **프롬프트를 더 구체적으로 쓸수록 결과가 좋았을 것이다.**
  "따뜻한 느낌"보다는 "크림색 배경(#FFFDF7), amber 포인트 컬러" 처럼 구체적인 디자인 토큰을 제공했다면 수정 횟수를 줄일 수 있었을 것이다.

- **검증 워크플로를 자동화했으면 더 수월했을 것이다.**
  검증을 전부 직접 브라우저를 열어 육안으로 진행했는데, Playwright MCP나 Chrome Extension 같은 도구를 연동했다면 AI가 스스로 화면을 확인하고 문제를 찾는 루프를 만들 수 있었을 것이다. "생성 → 자동 검증 → 수정"까지 하나의 흐름으로 이어졌다면 수정 반복 횟수를 크게 줄일 수 있었을 것이다.

### 배운 점

- 바이브 코딩의 핵심은 "무엇을 만들지 명확히 아는 것" 이다. AI는 구현을 가속화하지만, 기획과 판단은 여전히 사람의 몫이다.
- 전체 맥락을 한 번에 제공하는 것이 단편적인 요청의 반복보다 훨씬 좋은 결과를 낸다.
- 생성 후 검증 루프를 짧게 가져가는 것이 중요하다. "생성 → 확인 → 수정 프롬프트"의 반복 주기가 빠를수록 최종 결과물의 품질이 올라간다.
